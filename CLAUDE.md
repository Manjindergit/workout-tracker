# Workout Tracker

Mobile workout logger built with React Native / Expo. Single-user today; the data layer is designed so multi-user accounts and sync can be added later without a rewrite.

## Tech Stack

- React Native via Expo (managed workflow), TypeScript (strict)
- expo-router — file-based navigation
- expo-sqlite — local storage (accessed only through the repository layer)
- Zustand — app state
- Jest + React Native Testing Library — tests
- ESLint + Prettier — lint/format

## Commands

| Task | Command |
|---|---|
| Install deps | `npm install` |
| Dev server | `npx expo start` |
| Run on Android | `npx expo run:android` (or press `a` in the dev server) |
| Run on iOS | `npx expo run:ios` (macOS only) |
| Tests | `npm test` |
| Single test file | `npm test -- path/to/file.test.ts` |
| Typecheck | `npx tsc --noEmit` |
| Lint | `npx expo lint` |

Run typecheck + lint + tests before considering any change done.

## Folder Structure

(Target layout — created as the app is scaffolded.)

```
app/              # Screens & navigation (expo-router file-based routes)
src/
  components/     # Reusable UI components (dumb, props-driven)
  db/             # SQLite schema, migrations, repositories — the ONLY data-access layer
  hooks/          # Shared hooks (business logic lives here or in stores)
  stores/         # Zustand stores (e.g. active-workout session state)
  types/          # Shared domain types (Exercise, WorkoutSet, Workout)
  utils/          # Pure helpers (unit conversion, date math)
assets/           # Fonts, images, icons
agent_docs/       # Deep reference docs (schema decisions, sync design) — read when relevant
.claude/          # Claude Code config & skills
```

## Conventions

- TypeScript strict; no `any`, no `@ts-ignore` without a comment explaining why.
- Functional components + hooks only; no class components.
- Screens (`app/`) stay thin: layout + hook calls. Logic goes in hooks/stores; persistence in repositories.
- All persistence goes through `src/db/` repositories. Never call SQLite from components or hooks directly — this boundary is what makes future multi-user sync possible.
- Every domain record carries `id` (UUID), `createdAt`, `updatedAt` — required for future sync/merge.
- Store weights in kilograms and distances in meters internally; convert to the user's units only at display time.
- Schema changes = a new migration file in `src/db/migrations/`. Never edit an existing migration.
- Naming: PascalCase components, camelCase functions/variables, kebab-case route files.
- Tests colocated as `*.test.ts(x)` next to the code they cover.

## Off-Limits

- Never read or edit `.env*`, `secrets/`, or signing material (`*.jks`, `*.p8`, `*.p12`, `*.mobileprovision`, `*.pem`, `*.key`).
- Never hand-edit `android/` or `ios/` (generated by `expo prebuild`) — configure via `app.json` and config plugins instead.
- Never edit `package-lock.json` by hand; change dependencies via `npm install`.
- Never modify an already-committed migration — add a new one.
- Don't add native modules that require a custom dev client without flagging it first (it breaks the Expo Go workflow).
